# 运营生产数字人 (Operations Digital Assistant) - 项目文档

## 🤖 整体框架

本项目是一个基于 **Streamlit** 的生产运维智能化工具，旨在通过 AI 能力辅助排查支付业务中的复杂问题。系统通过不同的 Agent 协同工作，实现从底层日志检索、数据库明细查询到业务逻辑审计的自动化流转。

## 核心功能模块（持续接入中）

### 1. 核心调度 (main.py)

* **交互入口** ：提供 Web UI 界面，支持“交易问题定位”、“清算状态排查”、“立减客诉排查”三大模式。
* **流控逻辑** ：根据用户选择的模式，自动路由至对应的 API 调用链路，并展示 AI 分析出的根本原因与处理建议。

### 2. 日志分析专家 (log_agent.py)

* **日志检索** ：对接 LDM 日志库，根据 F11（跟踪号）、F33（前置机构）及时间窗口自动匹配原始报文。
* **AI 诊断** ：
* **核心提取** ：从海量日志中提取关键 `logId` 后的报错片段。
* **多模态分析** ：集成  **DeepSeek-V3/R1** ，结合 ISO8583 报文定义及历史解决经验（KB）进行根本原因分析。

### 3. 数据查询专家 (oracle_agent.py)

* **自动化 SQL 构建** ：支持多维度（F11, F22, F32, f33, F42, F100）动态拼装 SQL，查询 `szopr.tbl_daily_trans_temp_15` 近 15 天交易明细。
* **清算状态识别** ：通过 `sett_ind` 字段快速判断交易是否完成清算。
* **响应码库对接** ：关联 `szopr.qs_resp_rep_code` 表，自动翻译银联拒绝、发卡拒绝或前置拒绝的具体含义。

### 4. 营销排查专家 (marketing_agent.py)

* **双 Agent 审计架构** ：
* **Analyst Agent (ReAct模式)** ：通过 `[Thought] -> [Action] -> [Observation]` 路径，核对时间、卡 BIN、商户代码（MID）、受理机构（ACPT）及金额门槛。
* **Auditor Agent (审计模式)** ：负责对分析报告进行逻辑挑错，防止遗漏排除性条件或金额起跳点。
* **敏感信息处理** ：内置 SM3 脱敏算法，确保卡号等隐私数据在查询过程中的安全性。

## 🚀 未来改造目标

为了进一步提升数字人在复杂营销场景下的自动化排查能力，后续开发将聚焦于以下方向：

### 1. 外部数据源集成

* **报名信息对接** ：联系业务端（周涛）获取营销活动报名表数据。将用户是否报名、报名时间段等信息整合入排查链路，解决“未中奖因未报名”类客诉的快速闭环。

### 2. 营销 Skills 增强 (RAG/Tools)

* **海量参数检索能力** ：建立专门的 `Marketing Skills` 库，赋予 AI 智能查询海量 **ACPT (受理机构代码)** 与 **MID (商户号)** 白名单/黑名单的能力。
* **多跳逻辑优化** ：针对复杂的叠加活动，开发支持多步查询的检索工具，使 AI 能够自主决定何时查询活动定义、何时查询商户属性。
